# @Event Decorator

The `@Event` decorator defines a custom event that components can emit. This page documents the test coverage, patterns, generated components, and implementation status.

## Decorator Properties

- **bubbles**: if true, the event bubbles up through the DOM hierarchy
- **cancelable**: if true, the event can be canceled with `event.preventDefault()`
- **composed**: if true, the event can pass through shadow DOM boundaries

export const EventTestCasesTable = () => {
  // Get test case status from test coverage data
  const testCaseStatus = testCoverage.testCaseStatus || {};
  const items = testCoverage.items || [];
  
  // Build test case rows from testCaseStatus
  const testCaseRows = Object.entries(testCaseStatus)
    .sort(([a], [b]) => parseInt(a) - parseInt(b))
    .map(([id, status]) => {
      const caseItems = items.filter(item => item.caseIds && item.caseIds.includes(parseInt(id)));
      return {
        id: parseInt(id),
        desc: status.description || `Test Case #${id}`,
        types: `${status.componentCount} component${status.componentCount !== 1 ? 's' : ''}`,
        implemented: status.implemented || false,
        tested: caseItems.filter(item => item.tested).length,
        total: caseItems.length,
        section: status.implemented ? 'implemented' : 'missing',
      };
    });
  
  return <TestCasesTable rows={testCaseRows} previewCount={testCaseRows.length} />;
};

## Component Grouping

Components are grouped by **bubbles** mode under `/test/wdio/event/components/<bubbles|no-bubbles>/`. This organization reflects whether the event bubbles up through the DOM hierarchy or remains isolated to its target element. Additional test cases (Test Cases #3-4) are located in separate folders (`event-basic`, `event-custom-type`) to demonstrate real-world usage patterns.
